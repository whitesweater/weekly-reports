先给第一次读者的路线图：这份报告到底在讲什么

一句话：我们先有一个强规则系统 G02/G04，它学到了三类可解释知识；现在要验证这些知识能否从“训练后规则打分”升级为“神经网络训练过程中的结构化 prior”。
1 · 背景G02/G04 学到了什么？

它不是黑盒模型，而是从训练数据里沉淀出三条实际有效的规则机制：相关物品优先、热门物品代替、见过物品惩罚。

2 · 问题为什么不能直接继续用规则？

规则系统可解释但不可端到端学习，无法自动和 embedding、attention、sequence dynamics 对齐。

3 · 方法Scheme-I 怎么升级？

固定同一个 next-item recommender 底座，把 rule prior、Mamba context、Transformer attention、gated fusion 变成可验证结构。

4 · 实验怎么证明结构有效？

跑 7 个数据集 × 4 个模型变体，所有变体使用同一训练协议、同一划分、同一指标。

5 · 结果最后看什么？

看 NDCG@10 / HR@10 / FutureNDCG@10、winner count 和 ablation 分布，而不是只看单个模型的一次分数。

01

预备知识：G02/G04 原来的规则推荐系统是怎么做的

第一次看这份报告时，需要先知道我们升级的对象是什么：G02/G04 不是神经网络，而是一套基于用户历史序列、item 共现图、时间近因和候选校准的规则推荐器。更准确地说，它沉淀了三类知识：哪些相关物品应该优先、候选不够时用哪些热门物品代替、哪些用户已经见过的物品应该被惩罚或过滤。Scheme-I 做的是把这三类规则知识改造成可训练模型里的 prior / bias / fusion signal。

知识 1相关物品优先

从训练集共现图学习 history item → target item 的关联强度；用户历史中相关的候选 item 应该排在更前。

知识 2热门物品代替

当共现候选不足 Top-10，或者某些用户历史无法产生足够候选时，用训练集热门物品作为 fallback 补齐推荐列表。

知识 3见过物品惩罚

用户历史中已经出现过的 item 通常不应继续占据 Top-10 推荐位；G02/G04 通过 hard filter 或 soft penalty 惩罚 seen items。

Step 1构建共现图

只用 train split 扫描用户序列。如果 item a 经常和后续 item b 共同出现，就累积一条 a → b 的边权。

cooc(a,b) = count / idf / association strength
optional normalization
→
Step 2相关物品优先

预测时用用户 history 里的 item 查共现邻居；越相关的候选 item 得分越高，形成主推荐列表。

candidate(j) += Σ_i weight(i→j)
→
Step 3热门物品代替

如果相关候选不足 Top-10，就用训练集热门物品补齐；这个 fallback 不偷看 test target，只使用训练侧统计。

if len(rank) < 10:
  fill_with(popular_items)
→
Step 4见过物品惩罚

对用户历史里已经出现过的 item 做 hard filter 或 soft penalty，避免 seen items 浪费 Top-10 槽位。

if item in seen_history:
  remove or score *= exp(-penalty)
G02/G04 的本质

它不是单一共现表，而是一套可解释排序策略：先让相关物品优先，再用热门物品兜底，最后惩罚用户已经见过的物品。

h_j = Σ_i G_rule(i,j) · x_i
为什么必须 train-only？

规则图如果使用 valid/test 的未来交互，就会把答案泄漏给模型。因此 G02/G04 升级进神经网络时，规则 prior 只能从训练集统计。

fit_rule_graph(train)
never fit on valid/test future items
Scheme-I 如何继承它？

Scheme-I 不再让规则系统直接替代模型输出，而是把它作为 rule_prior_logits、attention/logit bias 或 gated fusion 的一个信号源。

neural_logits
 λ · rule_prior_logits
 optional learned gate
一句话连接旧方法和新方法： G02/G04 原来是“相关物品优先 + 热门物品代替 + 见过物品惩罚”的规则排序器；Scheme-I 是“固定一个神经 next-item recommender，再把这些规则机制变成可学习训练流程中的 prior / bias / fusion signal”。
02

从 Scheme-H 到 Scheme-I：我们到底改了什么

Scheme-H 已经证明“规则 prior 可以进入神经训练并被系统性消融”，但 naive residual-sum 会把不同坐标系的信号硬加在一起。Scheme-I 的核心改动是：不再堆分支，而是改变信号注入位置。

Rule prior

train-only 共现图不再作为 noisy branch 相加，而是进入 sparse item logit bias / attention-style prior。

Mamba / SSM

长程状态作为 context token 条件化注意力，避免与 Transformer hidden space 硬相加。

Transformer

保留自注意力作为最稳定 neural learner，用来学习细粒度 item interaction。

Gate

full 版本用 learned softmax gate，而非固定 residual sum，让模型自己决定各信号权重。

03A

共同基础模型：四个变体都挂在同一个 next-item recommender 上

这里的四个实验变体不是四个完全独立的模型，而是同一个轻量级序列推荐基础模型，加上不同结构模块开关。基础模型不是 ManCAR 原模型，也不是 NanoGPT 原版；它是在 clwm_rec/hybrid_model.py 中实现的 item-sequence next-item recommender。

模型任务

输入用户历史 item 序列，输出所有候选 item 的 ranking logits，用来预测下一个 item。

history = [item1, item2, ..., itemT]
model(history) → z_user
z_user · item_embeddingᵀ → logits over all items
为什么要固定基础模型？

如果每个变体底座不同，就无法判断性能差异来自 rule、Mamba、attention，还是来自底座变化。因此 Scheme-I 固定 pooling_mean_last scaffold，让实验只比较结构模块。

fixed_scaffold:
  base_encoder = pooling_mean_last
  always_enabled = true

Step 1：item embedding + position embedding

每个 item id 先查一个可学习向量，同时加上位置向量，让模型知道一个 item 在历史序列中的相对位置。

item_id → item_embedding
embedding_t = item_embedding_t + position_embedding_t
Step 2：mean-last pooling base encoder

基础模型同时取历史 item 的平均表示和最后一个有效 item 的表示。平均表示对应整体兴趣，最后一个 item 对应最近意图。

mean_emb = mean(history embeddings)
last_emb = last valid item embedding
z_base = MLP([mean_emb, last_emb])
Step 3：item logits head

用户向量 z 被投影回 item embedding 空间，再和所有 item embedding 做点积，得到全量 item vocabulary 上的排序分数。

query = W z
logits = query · item_embeddingᵀ + item_bias
rank_items = sort(logits)
03B

四个模块如何接到基础模型上

共同基础模型固定以后，Scheme-I 比较的就是不同模块如何改变用户向量 z 或最终 logits。下面这四个变体都共享 item embedding、position embedding、mean-last base encoder 和 dot-product item head。

Variant 1

transformer_only

基础模型加 Transformer self-attention。它检验纯神经 attention 在同一底座上能学到多少 item transition。

z_base = base(history)
z_attention = Transformer(history)
z = LayerNorm(z_base + z_attention)
Variant 2

rule_attention_bias

用户向量仍来自 base + attention，但规则 prior 不再变成 hidden branch，而是在最后 logits 上作为稀疏 bias 推一把。

z = LayerNorm(z_base + z_attention)
neural_logits = item_head(z)
logits = neural_logits + λ · rule_prior_logits
Variant 3

mamba_context_token

Mamba/SSM 不再和 Transformer 输出硬相加，而是先压缩历史成为 context token，再让 Transformer attend to this token。

context = Mamba(history)
tokens = [context_token, item1, ..., itemT]
z_attention = Transformer(tokens)
z = LayerNorm(z_base + z_attention)
Variant 4

gated_fusion_full

完整模型同时得到 base、attention、Mamba、rule 四个向量，用 learned softmax gate 决定每类信号权重。

z = α₁z_base + α₂z_attention
  + α₃z_mamba + α₄z_rule
α = softmax(gate_logits)
logits = item_head(z) + λ · rule_prior_logits
一句话总结： Scheme-I 的“基础模型”是 item embedding + position embedding + mean/last pooling 用户编码器 + item dot-product ranking head。 四个模块不是替换这个模型，而是在它上面测试 attention、rule prior、Mamba context 和 gated fusion 的不同接法。

研究问题能不能把规则推荐变成可训练结构？

原来的 G02/G04 是基于共现、时间衰减和邻居规则的推荐系统。现在的目标是把这些规则信号放进神经网络训练流程，而不是只做训练后的 lookup。

Evaluation

统一实验矩阵

同一训练脚本、同一划分、同一指标比较 7 datasets × 4 variants，避免“模型 A 和模型 B 评估口径不同”的问题。

数据如何处理

每个用户的行为先按时间排序，构造 history → next item 样本。规则共现图只从训练集统计，不接触 valid/test future item。

基础模型是什么

基础模型不是独立 baseline 名称，而是所有变体共享的 item-sequence encoder：embedding、position、mean-last pooling、MLP、item dot-product head。

如何 fusion

前三个变体控制单一结构因素；full 变体使用 learned softmax gate 把 base、attention、Mamba、rule 四类信号做可学习融合，而不是固定相加。

结论：结构修复有效，但 full fusion 仍需谨慎

Mamba context token

在 3/7 数据集按 NDCG@10 胜出，说明 context-token 位置比 residual branch 更合理。

Rule attention bias

在 2/7 数据集胜出，规则 prior 作为 bias 比并联 embedding branch 更稳。

Gated fusion full

只在 CDs_and_Vinyl 胜出，说明 full 结构仍可能过拟合或 gate 学习不足。

08

新增实验：把 attention bias / Mamba 迁移回原始 ManCAR

前面的 Scheme-I GPU-main 实验是在 clwm_rec 的结构化 hybrid scaffold 上验证 rule prior、Mamba context token、Transformer attention 和 gated fusion 的不同接法。当天我们追加了一个更贴近最终落地的问题：这些机制能不能不再停留在 scaffold 里，而是直接迁移进原始 ManCAR 主路径，并用 CLI flags 做同一个 7 个数据集 × 4 组开关的消融实验。

一句话：这次新增实验不是重新发明一个模型，而是把两个机制接进原 ManCAR：attention_bias 是 Transformer attention-score bias，mamba 是结构性 context token；默认全关时仍是 ManCAR baseline。
为什么要做这个实验

Scheme-I scaffold 证明结构方向有信号，但最终要回答的是“原 ManCAR 能不能直接支持这些机制”。所以这次实验固定原 ManCAR runner、数据集和训练流程，只切换两个新机制开关。

baseline = ManCAR(attention_bias=0, mamba=0)
ablation = same ManCAR + optional mechanisms
attention_bias 接在哪里

它不是 item-logit sparse bias，而是在 ManCAR Transformer 内部，把 Swing prompt prior 转成有限 additive score bias，并在 softmax 前加到 attention scores 上。

scores = QK^T / sqrt(d)
scores += padding_or_causal_mask
scores += λ · swing_attention_bias
Mamba 接在哪里

它不是额外并联分支，而是用本地 SSM/Mamba-style block 从历史 item embedding 生成一个 context token，再把它 prepend 到 ManCAR 的 Transformer 输入前缀里。

context = MambaSSM(history_embeddings)
tokens = [context] + prompt_tokens + history_tokens
1 · 需求澄清先锁语义

当天先明确两个机制都要迁移进原 ManCAR；attention_bias 必须是 Transformer attention-score bias，mamba 必须是 context-token 注入。

2 · 实现方式只用 CLI flags 开关

新增 --attention_bias、--attention_bias_weight、--mamba、--mamba_kernel，默认值保持 off/off。

3 · 安全检查先跑小验证

覆盖 parser defaults、zero-bias equivalence、KV-cache slicing、float additive mask、context token 不进入 prompt_latent 等行为。

4 · 主实验跑 main7×4

7 个 Amazon 数据集 × 4 组开关，共 28 个 ManCAR 训练/评估 cell；每个 cell 产出同一 run tag 下的确定性日志。

5 · 结果口径完成优先，不夸大性能

验收标准是 28/28 完整产出和可解析报告；这轮结果只说明迁移成功和局部信号，不声明机制稳定优于 baseline。
